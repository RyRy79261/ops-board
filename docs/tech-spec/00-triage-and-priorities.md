## 2. Per-board detail

### OpsBoard — Category Board (desktop) (`D3JA0i`) — P0 / high

PRIORITY P0: this is the default Category view — one of the three core views and the canonical app shell (header + 280px sidebar + main + tabs). It anchors the window-state and status models and the shared component kit/tokens, all explicitly P0 per the product brief.

COMPLEXITY high (not very-high): the board is structurally a single screen (no side-by-side screen variants), but it densely lays out the FULL TaskCard state matrix — 3 status states (done/not-started/in-progress) × 4 window states (open/closing/closed/not-yet) plus a blocked overlay with a dependency caption, across 5 category groups and 11 task instances — on top of 7 distinct structural surfaces (AppHeader, Sidebar, NavCard, MissionDetailHeader, StatTile, ProgressBar, ViewTabs, CategoryGroupHeader). The high count of state permutations and the heavy, multi-axis drift (task rows mirror Cmp·Card not TaskCard; bare pills vs bordered WindowStatePill; sharp vs pill tags; missing 44px touch target) push it to high.

DRIFT SUMMARY: the single most important reconciliation is that the board's task rows are named 'Card' and follow the Cmp·Card node tree (Inner>Status+Col) rather than the canonical TaskCard tree (Touch44>Cycle + Body), and they overload one bare 'Pill' frame to express window-state, blocked-status, and plain target dates — whereas the canon separates WindowStatePill / StatusBadge / CategoryTag. Verification must decide canonical identity and unify the pill/tag/touch-target treatments. Also flag: VoiceFAB + Toast (the P0 voice command + confirmation flow) are NOT present on this board and must be specced from the dedicated voice board.

FAN-OUT rationale: 8 structural surfaces + a 12-permutation card matrix → too much for 1-2 agents; the TaskCard matrix alone is a full agent. 3 agents (shell/nav, header+groups, TaskCard+state-matrix) cleanly partition the work with minimal overlap.

**Surfaces:**
- **App Shell / Top Header (AppHeader)** — Global desktop chrome: wordmark + (intended) global actions. Full-width bar above the body, muted fill with bottom border. _(states: 1)_
- **Sidebar — Missions list (Sidebar + NavCard)** — 280px left rail listing missions as navigable NavCards. 'MISSIONS' eyebrow header + scrollable mission list. _(states: 2)_
- **Mission Detail Header (MissionDetailHeader)** — Header of the main panel for the selected mission: title, target/cliff date, 4 stat tiles, and a segmented progress bar. _(states: 3)_
- **View Tabs (ViewTabs)** — Three-way view switcher between the core views. Underline-active tab pattern. _(states: 3)_
- **Category Group — MEDICAL (CategoryGroupHeader + task list)** — First category group: dot+label+count header over a stacked list of task cards. Demonstrates the widest range of card states. _(states: 5)_
- **Category Group — BUREAUCRATIC** — Second category group (cat-bureaucratic, count 0/3). Demonstrates BLOCKED, WINDOW CLOSED, and CLOSING states. _(states: 4)_
- **Category Group — TRAVEL** — Third category group (cat-travel, count 1/2). DONE + BLOCKED states with dependency line. _(states: 3)_
- **Category Group — GEAR** — Fourth category group (cat-gear, count 0/1). Demonstrates the IN-PROGRESS / partial status box. _(states: 2)_
- **Category Group — TECH** — Fifth category group (cat-tech, count 0/1). Demonstrates the NOT-YET (not-yet-open window) state. _(states: 2)_
- **TaskCard window-state matrix (aggregate, derived across all groups)** — Across the five groups the board lays out the complete TaskCard state matrix that downstream specs must enumerate as one component's variants. _(states: 8)_

**Recommended spec-agent split:**
- `App shell + navigation chrome` — Spec the desktop shell composition: AppHeader (with the missing Actions-cluster decision), Sidebar (280px) + NavCard active/selected state and cliff-chip, ViewTabs (active/inactive, 3 views, navigation behavior). Cover layout grid (header / 280px sidebar / fill main), borders, muted-surface tokens. Resolve AppHeader-actions and ViewTabs stray-stroke drift.
- `Mission detail header + category group scaffolding` — Spec MissionDetailHeader (Title, Target date format, 4× StatTile, segmented ProgressBar) and CategoryGroupHeader (Dot/Label/Count, the 5 category color tokens cat-medical/bureaucratic/travel/gear/tech), plus the Content list layout (group gaps, ordering, per-group counts). Reconcile StatTile fontSize/nesting drift and Target ISO-vs-human date drift; define group-to-color mapping.
- `TaskCard component + full window/status/blocked state matrix` — Author the canonical TaskCard spec from this board's 11 flattened 'Card' instances: status indicator (DONE/NOT_STARTED/IN_PROGRESS, the 44px Touch44 tap-to-cycle target, glyph-vs-shape), CategoryTag (sharp-vs-pill radius), WindowStatePill semantics (OPEN/CLOSING/CLOSED/NOT_YET) AND the overloaded BLOCKED + dependency 'blocked by' caption, plus the left-accent border + opacity dimming recipes. This is the densest, highest-drift surface and warrants its own agent. Must reconcile TaskCard-vs-Cmp·Card identity and note the absent VoiceFAB/Toast voice flow as an out-of-board integration dependency.

**Drift candidates (15):**
- **Task row node named 'Card' (×11) vs canonical TaskCard** — PRIMARY DRIFT. Board task rows are flattened Cmp·Card trees: Card > Inner > [Status(18x18) + Col(TaskName, MetaRow(Tag, Pill[, Dep]))]. Canonical TaskCard is: TaskCard > [Touch44(44x44) > Cycle(18x18 box) , Body(Name, Meta(Tag, Win))]. Different node names (Inner/Col vs Body, Status vs Touch44>Cycle, Win vs Pill), missing Touch44 wrapper, extra Dep line. Verification must reconcile which component these rows actually are.
- **Status box missing 44px touch target** — Canonical TaskCard wraps the 18x18 cycle box in a 44x44 'Touch44' frame (the tap-to-cycle hit area — the board's only interaction). Every board 'Status' box (×11) is a raw 18x18 frame with no 44px wrapper. Accessibility/interaction-spec gap to reconcile.
- **Status-box fill glyphs diverge from StatusCycleButton** — Board DONE box = filled $success bg + child text '✓'; IN_PROGRESS box = #ff6b351f bg + primary '◼' (fontSize 9); empty box = border-only. Canonical StatusCycleButton has a single '◼' glyph (fontSize 8) and no fill-state variants; canonical TaskCard's Cycle has a child rectangle Fill (8x8) rather than a text glyph. Board encodes status via text glyphs, canonical via shape — reconcile glyph vs shape and the DONE-fill variant.
- **CategoryTag rendered sharp-cornered (cornerRadius 0)** — All 11 board 'Tag' frames have no cornerRadius (board radius token = 0). Canonical CategoryTag is cornerRadius 999 (pill). Either the global radius=0 token overrides canonical, or the canonical pill is wrong. Same applies to NavCard cliff chip vs board (chip kept 999). Inconsistent radius usage on this board.
- **WindowStatePill rendered as bare frame, not a bordered pill** — Board 'Pill' frames have no stroke/border/cornerRadius — just icon+text inline. Canonical WindowStatePill has stroke=$border + cornerRadius 999. Board's pill also overloads three semantics through one shape: window-state (CLOSING/WINDOW CLOSED/NOT YET), blocked-status (BLOCKED + triangle-alert), and plain target date (no icon). Canonical separates WindowStatePill from StatusBadge.
- **OPEN window-state has no pill at all** — DONE/OPEN cards render only a plain date string ('15 Mar', '1 Apr', '20 Mar') with no icon and no 'OPEN' label. Canonical WindowStatePill default = circle icon + 'OPEN' text. Board collapses OPEN into a bare date. Spec must define whether OPEN shows a pill or just a date.
- **BLOCKED state conflated into the window-state pill + dimming + Dep line** — BLOCKED is not a window state in the product model (window = open/closing/closed/not-yet) but the board renders BLOCKED through the same 'Pill' slot (triangle-alert + 'BLOCKED'), plus card opacity 0.6 and an extra 'Dep' text line (⚠ blocked by: …). No canonical component represents the inline dependency line on a task card — this is net-new (resembles a DependencyConnector/Dep caption). Reconcile BLOCKED as status vs window, and spec the dependency caption.
- **WINDOW CLOSED card uses muted fill + different border token + opacity 0.55** — The closed bureaucratic card uses fill=$muted (not $card), stroke=$muted-foreground (not subtle), opacity 0.55 — a distinct visual recipe from the other dimmed (0.6) cards. Three different dim/fill recipes across closed/blocked/not-yet need a single canonical mapping.
- **Left-accent border encoding (strokeWidth {left:2})** — CLOSING/CLOSED/BLOCKED/NOT-YET cards use a left-only 2px colored border (warning / muted-foreground / subtle) while DONE/OPEN cards use a full 1px $border. Canonical TaskCard/Cmp·Card use a uniform 1px border with no left-accent concept. This left-accent state-encoding is net-new board behavior to capture.
- **AppHeader missing Actions cluster** — Board header has only the Wordmark; canonical AppHeader includes Actions (SyncStatus, Search, Notifications, More). space_between layout leaves the right side empty. Either the desktop header intentionally drops actions or this is incomplete.
- **StatTile fontSize + nesting drift** — Board stat values are fontSize 22 in bespoke DONE/BLOCKED/CLOSING/TOTAL frames; canonical Cmp·StatTile Val is fontSize 30 and is its own reusable. MissionDetailHeader canonical references StatTiles by name; board inlines them. Also MissionSummaryCard (a separate canonical) duplicates this stat-block at fontSize 20 — three sizes for the same stat block across the system.
- **MissionDetailHeader target string format** — Board Target = 'Target: 27 Apr 2026' (sentence case, human date); canonical MissionDetailHeader Target = 'TARGET: 2026-04-27' (uppercase, ISO). Copy/format drift in the cliff date presentation.
- **ViewTabs inactive tabs carry stray stroke:$primary** — Board inactive tabs (TIMELINE, DEPENDENCIES) include stroke='$primary' with no strokeWidth, so no visible border but a dangling prop; canonical inactive 'Tab' has no stroke. Likely extraction noise but flag for clean-up.
- **No VoiceFAB / RecordingPanel / Toast present on this board** — The voice command surface (VoiceFAB) and its toast/confirmation flow — a P0 part of the product — are absent from this desktop board. The Category view in production must compose VoiceFAB + Toast; this board does not show them, so their spec must be sourced from the dedicated voice board, not here. Flag the integration gap.
- **GroupHeader child names GLabel/GCount** — Board uses GLabel/GCount; canonical CategoryGroupHeader uses Label/Count. Cosmetic naming drift, low risk but note for codegen consistency.

### Timeline view (desktop) (`a3Dgz`) — P0 / high

PRIORITY P0: Timeline is one of the three core MVP views called out in the brief; it is the canonical demonstration surface for the window-state model (too_late_by cliff -> WINDOW CLOSED / THIS WEEK / IN Nd / NO CLIFF) and the read-only status-cycle interaction. Non-negotiable for MVP.

COMPLEXITY high (not very-high): the file is 244 nodes/126KB but the structure is repetitive — 6 buckets of near-identical horizontal task rows. The real complexity is the STATE MATRIX, not novel surfaces: this single board exercises the entire window-state taxonomy (4 future/past states + no-cliff + closed-dimmed) crossed with all task statuses (not-started/in-progress/done/blocked/not-yet) and all 5 categories, PLUS a full page shell (chrome, sidebar w/ active+inactive missions, mission detail summary, view tabs). That breadth of states + the heavy, systematic drift from canonical components (the bespoke horizontal TaskCard, overloaded window/status Pill, dot+icon CategoryTag, missing 44px touch target) is what pushes it to high. Not very-high because there are no truly novel interactive organisms here (no voice/recorder/disambiguation flows live on this board).

FAN-OUT 2 agents: distinct-surface count is moderate (page shell cluster + the timeline view cluster) and they cleanly separate along a natural seam — Agent A owns the persistent chrome/mission-context that is shared with the Category/Dependencies boards, Agent B owns the Timeline-specific bucket+task-row state matrix and the bulk of the drift. One agent would overload on the task-row state matrix; three would over-fragment the shared shell. 2 is the right balance for a high-complexity, single-view board.

KEY DRIFT THEME for downstream bidirectional verification: this board's task row is a fundamentally different (horizontal) tree from the canonical (vertical) TaskCard, and it MERGES status + window-state into one pill while the canonical kit keeps StatusBadge / WindowStatePill / CategoryTag / StatusCycleButton as separate atoms. Reconciliation must decide the source of truth before build. Also note the brief explicitly mandates 'never overdue -> WINDOW CLOSED (muted grey + strikethrough)' and a 44px touch target — the board honors the wording (WINDOW CLOSED, opacity dim) but omits the 44px hit area and uses opacity rather than strikethrough.

**Surfaces:**
- **Top Header (app chrome)** — Persistent 52px top bar spanning full width. Holds only the OPS/BOARD split wordmark (mono, primary + muted). Bottom hairline border. Provides app identity; no nav controls, voice FAB, or sync indicator rendered here on this board. _(states: 1)_
- **Sidebar — Missions list** — 280px left rail listing the user's missions so one can be selected as the active detail context. Header row 'MISSIONS' eyebrow + count '04'. Each mission row is a NavCard: left accent bar (primary when active, background when inactive), mission name (DM Sans 14/500), and a meta row with a window-summary chip + done/total task count. _(states: 3)_
- **Mission Detail header (bespoke MissionSummaryCard / MissionDetailHeader)** — Top of the main column. Identifies the active mission and summarizes its health. Title row = live primary dot + 'AfrikaBurn 2026' (DM Sans 25/600). Target line = mono 'TARGET:' + date '2026-07-29' + separator + location 'TANKWA KAROO, ZA'. Stats row of 4 stat tiles (val over label). Three-segment progress bar. _(states: 2)_
- **View Tabs** — Segmented tab strip switching the main column between the 3 core views. Bottom hairline border; active tab carries a 2px primary bottom underline + foreground label. _(states: 3)_
- **Timeline — bucket list (the primary view)** — Vertical stack of ISO-week buckets ordered by too_late_by cliff (past -> future -> no-cliff). Demonstrates the full window-state taxonomy across weeks. Each bucket = a TimelineWeekHeader (range label + days-until/state pill + horizontal rule) over a vertical stack of task cards. _(states: 6)_
- **Timeline task row (bespoke flattened TaskCard) — window/status variant matrix** — The atomic row inside each bucket. Horizontal layout: status box (left, tappable StatusCycleButton role) + NameBlock (task name, optional blocked-by hint sub-line) + CategoryTag (dot + lucide icon + mono category label) + right-aligned window/status Pill. ~10 instances laid out side by side across buckets exercise every window-state × status combination. _(states: 7)_

**Recommended spec-agent split:**
- `Agent A — Page shell & mission context (chrome + sidebar + detail header + tabs)` — Spec the desktop Timeline page frame (1320x1160, vertical: 52px Top Header / Body Row = 280px Sidebar + fluid Main Column). Cover: Top Header/AppHeader (wordmark, omitted affordances vs canonical AppHeader — flag scope gap); Sidebar (MISSIONS eyebrow + '04' count, 4 NavCard mission rows: active vs inactive, accent bar, window-summary chip taxonomy '1 CLOSING'/'T-12d', done/total count); Mission Detail header (live dot + title + target line w/ location, 4 StatTiles DONE/BLOCKED/CLOSING/TOTAL, 3-segment ProgressBar) reconciling MissionSummaryCard vs MissionDetailHeader; ViewTabs (3 tabs, active=Timeline, underline/label states). Document every drift candidate in this region against canonical Sidebar/MissionDetailHeader/ViewTabs/AppHeader and the brief's window-summary chip spec.
- `Agent B — Timeline buckets & task-row state matrix (the core view)` — Spec the Timeline organism: bucket ordering by ISO week of too_late_by (past-closed -> this-week -> future -> NO CLIFF trailing). For TimelineWeekHeader/DaysPill spec all pill states (WINDOW CLOSED, THIS WEEK [primary], IN Nd, UNSCHEDULED) + range label + rule. For the bespoke horizontal TaskCard spec the FULL state matrix: status (not-started, in-progress=primary-square, done=success-check, blocked, not-yet/locked=STARTS, window-closed dimmed opacity 0.6), the merged window/status Pill (DONE/BLOCKED/CLOSING·T-2D/STARTS DATE/NO CLIFF/WINDOW CLOSED, stroke recolor), CategoryTag variants (medical/bureaucratic/travel/gear/tech with dot+icon+label), the blocked-by hint sub-line, and StatusCycleButton (note missing 44px touch target). Enumerate the ~10 card instances and tie each to a window-state × status combination. Capture all TaskCard/StatusCycleButton/CategoryTag/WindowStatePill/TimelineBucket drift candidates for downstream verification.
