# OpsBoard ‚Äî Product Brief v3

**Voice-first personal mission planner with dependency-aware task orchestration.**
**Built on the Camp 404 monorepo conventions, reusing its voice + MCP + DB infrastructure.**

> **What changed from v2:** v2 specified building a voice pipeline and an MCP server from scratch. After reviewing the `ryry79261/camp-404` monorepo, that's redundant ‚Äî Camp 404 already has a working Groq Whisper transcription pipeline, a full MCP OAuth server with a per-domain tool pattern, a versioned AI-prompt package, a Drizzle/Neon data layer with strict migration discipline, and a shared shadcn UI package. This brief re-targets OpsBoard to **adopt those conventions and extend those assets** rather than rebuild them. The product thinking from v2 (the "too late by" temporal model, the read-only-dashboard philosophy, the friction analysis) is unchanged and carried forward.

---

## 0. Placement ‚Äî DECIDED: standalone repo

**Target repo: `github.com/RyRy79261/ops-board`** (currently empty ‚Äî one-line README, single initial commit; this is a greenfield scaffold).

OpsBoard is a **single-user personal tool**; Camp 404 is a **multi-user camp app** (30‚Äì80 members) with rank/approval/team gating and a captain/team/consent MCP scope model. That substrate is wrong for a personal tool, so OpsBoard lives in its **own repo and mirrors Camp 404's conventions** rather than living inside the camp-404 monorepo.

Concretely:
- New `ops-board` Turborepo + pnpm monorepo with the same layout and the same `@opsboard/{ui,db,types,ai-prompts,core}` package split (see ¬ß1).
- **Lift** the voice pipeline components and the MCP OAuth scaffolding from Camp 404 as the starting point (specific files named in ¬ß3, ¬ß4).
- **Strip** the camp-specific auth/rank/team/consent machinery down to single-user (or no) auth ‚Äî keep the OAuth+PKCE shell and audit logging, drop the captain/team scope resolution.

The conventions and concrete code assets are the valuable part; the shared multi-user substrate is not carried over.

## 1. Monorepo Layout & Conventions (from Camp 404)

Turborepo + pnpm workspaces. **Node >= 22, pnpm 10.x.** Mirror this structure:

```
apps/
  web/        Next.js 16 app (App Router, React 19, Tailwind v4)
  mobile/     Capacitor host wrapping the web static export (defer ‚Äî see ¬ß8)
  admin-cli/  Node CLI for data ops (optional for OpsBoard)
packages/
  ui/         Shared shadcn/ui components            (@opsboard/ui)
  db/         Drizzle schema + migrations            (@opsboard/db)
  types/      Zod schemas + shared TS types          (@opsboard/types)
  ai-prompts/ Versioned prompt templates             (@opsboard/ai-prompts)
  core/       Pure business logic, no I/O            (@opsboard/core)
  eslint-config/  typescript-config/
```

Workspace + tooling files to copy and rename: `pnpm-workspace.yaml` (`apps/*`, `packages/*`), `turbo.json`, root `package.json` scripts (`build`/`dev`/`lint`/`typecheck`/`test`/`format` all fan out via `turbo run`), `.prettierrc.json`, `.npmrc`, `eslint-config`, `typescript-config`, and an `AGENTS.md` adapted from Camp 404's (it's an excellent template for the agent rules ‚Äî keep the DB discipline, env-var, and conventions sections).

Commands (identical to Camp 404):
```bash
pnpm install                                  # --frozen-lockfile in CI
pnpm dev                                       # dev servers
pnpm turbo run lint typecheck test build       # the full CI gate
pnpm --filter @opsboard/web dev                # per-package work
```

**Conventions to carry over verbatim:**
- TypeScript throughout; shared types + Zod schemas in `@opsboard/types`; validate external input at the boundary with Zod.
- Pure logic (dependency-graph resolution, "blocked" computation, window-state derivation) lives in `@opsboard/core` with unit tests ‚Äî keep it I/O-free so it's trivially testable, exactly like Camp 404's `core/access.ts`, `core/family-tree.ts`.
- Prefer editing existing files; don't add abstractions a task doesn't need.
- Add/update Vitest tests with behavioural changes.
- **Env-var discipline:** when adding an env var, update **both** `.env.example` **and** the `globalEnv` array in `turbo.json` ‚Äî miss the latter and Turbo's cache won't invalidate on change, causing stale builds.

## 2. Data Layer (Drizzle + Neon ‚Äî follow Camp 404's discipline exactly)

NeonDB Postgres + Drizzle ORM. **`packages/db/src/schema.ts` is the single hand-authored source of truth.** Everything under `packages/db/migrations/` is generated by drizzle-kit ‚Äî **never hand-edit a migration.**

Workflow (identical to Camp 404):
1. Edit `packages/db/src/schema.ts`.
2. `pnpm --filter @opsboard/db db:generate` ‚Äî appends a new numbered migration, updates journal/snapshot.
3. Commit the generated migration with the schema change.
4. `pnpm --filter @opsboard/db db:migrate` applies it. Migrations are **append-only**; `0000_*` is frozen.

**Two drivers** (copy Camp 404's `packages/db/src/index.ts` pattern):
- `createHttpDb()` ‚Äî stateless, **no transactions**, for route handlers / server components.
- `createPooledDb()` ‚Äî WebSocket pool, **supports transactions**, for the CLI / any multi-statement atomic work (e.g., creating a mission with seeded tasks in one transaction).

**Bespoke over generic** (Camp 404's explicit stance): distinct domain tables, no generic key-value/CMS store. OpsBoard's domain is small enough that this is natural.

### Schema (in Drizzle, conceptually)

```
missions
  id            uuid pk
  name          text not null
  target_date   date            -- the fixed real-world event date
  created_at, updated_at  timestamptz

categories
  id            uuid pk
  name          text not null
  slug          text not null unique
  color         text not null   -- OKLCH or hex; align to @opsboard/ui tokens
  sort_order    integer default 0
  is_default    boolean default false
  -- seed: medical, bureaucratic, travel, gear, tech

tasks
  id            uuid pk
  mission_id    uuid ‚Üí missions(id) on delete cascade
  name          text not null
  category_id   uuid ‚Üí categories(id)
  status        text default 'not-started'
                  check in ('not-started','in-progress','done')
  too_late_by   date            -- "after this date the task is moot" (NOT a due date)
  not_before    date            -- "can't start until this date"
  notes         text
  sort_order    integer default 0
  created_at, updated_at  timestamptz

task_dependencies
  id            uuid pk
  task_id       uuid ‚Üí tasks(id) on delete cascade
  depends_on_id uuid ‚Üí tasks(id) on delete cascade
  unique(task_id, depends_on_id)
  check (task_id <> depends_on_id)
```

**`blocked` is NOT stored.** It's computed from the dependency graph at read time: a task is blocked if any dependency's status ‚âÝ `done`. Put that derivation (and cycle detection, critical-path, window-state) in `@opsboard/core` as pure functions. Stored statuses are only `not-started | in-progress | done`.

**Temporal model (the v2 correction, unchanged):** tasks carry a `too_late_by` cliff, not a due date. The UI communicates window state ‚Äî *open / closing / closed / not-yet* ‚Äî never "overdue." See ¬ß6.

**Security:** OpsBoard holds no passports/IDs/bank details, so the pgcrypto column-encryption Camp 404 uses for POPIA isn't needed here. If that ever changes, follow Camp 404's `packages/db/src/crypto.ts` + `id-documents.ts` pattern (encrypt in the route handler, never store plaintext).

## 3. Voice ‚Äî Extend the Existing Pipeline (don't rebuild)

Camp 404 already has a **working browser‚ÜíGroq transcription pipeline.** Reuse it wholesale; OpsBoard's only new work is the **intent‚Üíaction layer**, which Camp 404 has scaffolded but not wired for commands.

### What already exists (copy from Camp 404 `apps/web`)
- `components/voice/recorder-panel.tsx` ‚Äî the mounted dictation surface (circular record button, live waveform, mm:ss timer, status label, error line). Stays open after each clip for repeat capture.
- `components/voice/use-voice-recorder.ts` ‚Äî the cross-browser `MediaRecorder` state machine (`idle | requesting | recording | processing | error`), MIME selection (the iOS-Safari `audio/mp4` fallback is the critical gotcha ‚Äî keep it), 2-min auto-stop, analyser wiring, unmount-safe cleanup.
- `components/voice/waveform.tsx` ‚Äî canvas waveform painter.
- `app/api/voice/transcribe/route.ts` ‚Äî Node-runtime POST: auth gate, dual rate-limit (per-user + per-IP), multipart parse, `audio/*` + size validation, server-side `promptKey` allow-list, calls Groq, scrubs error leakage.
- `lib/groq.ts` ‚Äî lazy Groq client; `transcribeAudio(file, {prompt})` ‚Üí `whisper-large-v3-turbo`, `response_format: "json"`, `temperature: 0`. (~216x realtime, $0.04/audio-hour.)
- `lib/voice-prompts.ts` ‚Äî domain-biasing Whisper prompt (jargon/names/numerals). Write an OpsBoard equivalent (mission names, "AfrikaBurn", "cardiology", "Envivas", "VFS", category words) ‚Äî Whisper truncates prompts at ~224 tokens, so keep it short and dense with rare terms.
- `lib/rate-limit.ts` ‚Äî in-memory token-bucket limiter.

### The gap OpsBoard fills: transcript ‚Üí structured action ‚Üí execute
Camp 404's transcribe route only **returns text** to be appended to a field. It has the *classifier scaffolding* but no command-execution route. Build that, following the existing pattern:

**Intent schema** ‚Äî extend the pattern in `@camp404/types/voice-intent.ts` (a Zod `discriminatedUnion("intent", [‚Ä¶])`). Create `@opsboard/types/voice-intent.ts` with OpsBoard intents:
```
create_mission        { name, targetDateHint? }
create_task           { missionHint?, name, categoryHint?, tooLateByHint?, notBeforeHint?, dependsOnHints?[] }
update_task_status    { taskHint, status: 'done'|'in-progress'|'not-started' }
update_task           { taskHint, field, value }
add_dependency        { taskHint, dependsOnHint }
remove_dependency     { taskHint, dependsOnHint }
delete_task           { taskHint }
delete_mission        { missionHint }
query                 { question }      // natural-language, answered from computed state
unknown               { rawTranscript }
```

**Intent prompt** ‚Äî mirror `@camp404/ai-prompts/voice-intent.ts` (a `{ system, user }` const). Create `@opsboard/ai-prompts/voice-intent.ts`. The system prompt: classifies + extracts fields, fuzzy-matches `*Hint` fields against current state (injected at call time), resolves relative time ("a week before the trip" ‚Üí `target_date ‚àí 7`; "mid-March" ‚Üí `2026-03-15`), and ‚Äî copying Camp 404's safety rule ‚Äî **destructive/ambiguous actions (delete_*, anything low-confidence) must be confirmed before execution, never assumed.**

**New command route** ‚Äî `app/api/voice/command/route.ts` (parallel to `transcribe/route.ts`, reusing its auth/rate-limit/validation guards):
1. Transcribe the clip via the existing `transcribeAudio()` (with the OpsBoard Whisper prompt).
2. Send transcript + a compact snapshot of current missions/tasks to **Claude** (intent parse). Use a pinned model ‚Äî Camp 404 pins **Claude Haiku 4.5** for cheap/fast classification and **Opus 4.8** for heavier work; intent parsing is a Haiku-class job.
3. Validate Claude's JSON against the `VoiceIntent` Zod union.
4. Execute via the same service functions the REST routes use (`@opsboard/core` + `@opsboard/db`).
5. Return `{ transcript, intent, result, confirmation }`. For low-confidence/destructive intents, return a `needsConfirmation` flag instead of executing.

**Model/prompt discipline (Camp 404 rule):** model IDs and prompt templates in `@opsboard/ai-prompts` are **pinned and versioned** ‚Äî never swap a model or edit a prompt in place; bump the version. Whisper STT is `whisper-large-v3-turbo`.

### Voice UI for OpsBoard
The camp `RecorderPanel` is built for *dictation into a text field* (it appends transcripts). OpsBoard needs a **command** affordance: a floating mic button (bottom-right desktop / thumb-reachable bottom-center mobile) that reuses `use-voice-recorder.ts` unchanged, but on transcript routes to `/api/voice/command` and surfaces the `confirmation` (or a tap-to-confirm disambiguation) as a toast ‚Äî rather than appending text anywhere. This is the single biggest piece of net-new UI; everything beneath it is lifted.

## 4. MCP ‚Äî Add an OpsBoard Tool Module (don't stand up a new server)

Camp 404 has a complete MCP server: `app/api/mcp/[transport]/route.ts`, a full OAuth 2.0 + PKCE flow (`oauth/authorize`, `oauth/register`, `oauth/token`, well-known metadata), a consent screen, a per-call scope resolver, and an audit log. Tools are organised **one file per domain** under `lib/mcp/tools/` and registered in `lib/mcp/server.ts`.

**The tool pattern to follow** (from `lib/mcp/tools/teams.ts`):
```ts
export function registerOpsboardTools(server: McpServer): void {
  server.registerTool(
    "get_blocked_tasks",
    {
      title: "List blocked tasks",
      description: "Tasks whose dependencies aren't all done yet, with blocker details.",
      inputSchema: { missionId: z.string().optional() },
    },
    async (args, extra) =>
      runTool({
        toolName: "get_blocked_tasks",
        extra,
        argsForAudit: args,
        handler: async () => {
          const db = createHttpDb();
          /* query + @opsboard/core blocked-derivation */
        },
      }),
  );
  /* ‚Ä¶more tools‚Ä¶ */
}
```
Then add `registerOpsboardTools(server)` to the server's registration function (Camp 404's is `registerCampMcpTools`). Every handler wraps in `runTool({ toolName, extra, argsForAudit, handler })` for audit logging + scope.

**Tools to expose** (each in `lib/mcp/tools/opsboard.ts`, or split missions/tasks if it grows):
```
list_missions
get_mission            { missionId | nameHint }
create_mission         { name, targetDate? }
update_mission         { missionId, name?, targetDate? }
delete_mission         { missionId }
list_tasks             { missionId?, category?, status?, blocked? }
create_task            { missionId, name, category?, tooLateBy?, notBefore?, dependsOn[]?, notes? }
update_task            { taskId, name?, category?, status?, tooLateBy?, notBefore?, notes? }
delete_task            { taskId }
add_dependency         { taskId, dependsOnId }
remove_dependency      { taskId, dependsOnId }
get_blocked_tasks      { missionId? }                 -- computed, with blocker names
get_closing_windows    { missionId?, daysAhead? }     -- tasks whose too_late_by is near
get_critical_path      { missionId }                  -- longest dependency chain to target
```

**Auth/scope:** single-user, so the camp's captain/team/consent scope model is overkill ‚Äî strip `lib/mcp/scope.ts`/`consent.ts` down to a single-principal check, but **keep** the OAuth+PKCE shell, the `claude.ai`/`anthropic.com` redirect-URI allow-list, `runTool` audit logging, and the per-call freshness pattern.

Result: "What's blocking me on AfrikaBurn?", "Mark the cardiology follow-up done", "What's about to close this week?" all work from Claude chat via MCP ‚Äî same as the voice commands, different transport.

## 6. Dashboard (read-only display ‚Äî v2 philosophy, Camp 404 styling)

The UI is a **status display, not an input surface.** All creation/editing happens via voice or MCP. The one direct interaction kept on the board: tapping a task cycles its status (`not-started ‚Üí in-progress ‚Üí done`).

Use the v1 HTML prototype (`ops-board.html`) as the visual reference, re-implemented with `@opsboard/ui` (shadcn) + Tailwind v4 tokens. Three views ‚Äî **Category**, **Timeline**, **Dependencies**. Plus the floating voice button (¬ß3) and a toast area for confirmations.

**Window-state language (not "overdue"):**
- *Window open* ‚Äî actionable, cliff far ‚Üí calm.
- *Window closing* ‚Äî `too_late_by` near ‚Üí amber.
- *Window closed* ‚Äî past `too_late_by`, task moot ‚Üí greyed, struck through.
- *Not yet* ‚Äî has `not_before` in the future, or is blocked ‚Üí dimmed with reason (lock icon / blocker names).

Derive all four states in `@opsboard/core` so the dashboard, voice `query`, and MCP queries share one source of truth.

## 7. Build Order

1. **Scaffold** the `ops-board` monorepo from Camp 404's skeleton (workspace, turbo, tsconfig/eslint, AGENTS.md) into the existing empty repo.
2. **DB:** schema.ts (missions, categories, tasks, task_dependencies), generate `0000` migration, seed categories, two-driver `index.ts`. Pure derivations (blocked, window-state, critical-path, cycle detection) in `@opsboard/core` with Vitest.
3. **REST routes + dashboard:** CRUD routes (`createHttpDb`), then port the prototype to React/Tailwind with `@opsboard/ui`, wire the three views + status-cycle tap.
4. **Voice:** copy the `components/voice/*` + `lib/groq.ts` + `lib/rate-limit.ts` assets; write OpsBoard Whisper prompt; build the `VoiceIntent` union (`@opsboard/types`) + intent prompt (`@opsboard/ai-prompts`) + `/api/voice/command` route; add the floating mic button + confirmation toast + disambiguation.
5. **MCP:** lift the OAuth shell, strip scope to single-user, add `tools/opsboard.ts`, register it, test via Claude.ai connector.
6. **Polish:** prompt tuning/edge cases, mobile swipe gestures, optional window-closing reminders.

## 8. Deferred / Known Traps (learned from Camp 404)

- **Capacitor mobile export is hard on Next 16.** Camp 404's `build:mobile` is currently broken and deferred ‚Äî `output: "export"` requires every route/page to be statically pre-renderable, but server-only route handlers and cookie-reading pages aren't. If OpsBoard wants native, plan for client-side screens calling a separately deployed API, and treat mobile as a later phase, not MVP.
- **Whisper iOS-Safari MIME gotcha:** keep the `audio/mp4` fallback in `use-voice-recorder.ts`; hardcoding `audio/webm` makes iOS silently fail.
- **Whisper prompt length:** ~224-token truncation ‚Äî keep the domain prompt short and dense.
- **MCP POST redirects:** Camp 404 uses meta-refresh + JS, never 302, because CSP `form-action 'self'` drops cross-origin 302s from POST handlers. Keep that if you lift the OAuth flow.
- **Origin resolution for MCP:** derive from request headers / `MCP_PUBLIC_URL`, never `VERCEL_URL` (which can point at the SSO-gated deploy URL).
- **Cron cap:** Vercel's daily-cron limit matters only if window-closing reminders are added; gate any `/api/cron/*` behind `Authorization: Bearer ${CRON_SECRET}` like Camp 404.
